# Unified Verification Suite

Run all project verification checks and produce a combined gap analysis report.

**Output:** `specs/4-docs/verification-report_{YYYY-MM-DD}.md`

## Schema check

This command targets schema version **2**.

Active project state:

!`"${CLAUDE_PLUGIN_ROOT}/scripts/schema-check.sh"`

Decide based on the output above:
- `STATUS=OK` — proceed.
- `STATUS=MISMATCH` or `STATUS=LEGACY` — orchestrators are sensitive to schema drift because their subcommands depend on layout. Recommend `/shit:migrate` before running. Refuse to proceed unless the user explicitly requests best-effort.
- `STATUS=UNINITIALIZED` — tell the user to run `/shit:init` first.

## What This Command Does

The verification suite runs three classes of checks:

1. **Universal checks** — checks that apply to every project regardless of language or framework (build health, test health, coverage, provenance coverage, TODO scan, unwired code analysis, dependency freshness, file complexity)
2. **Pipeline traceability checks** — graph-completeness checks across the reification chain (delegated to other plugin commands when their reports exist)
3. **Gate-defined checks** — project-specific structural and architectural checks read from `specs/gates/verify-0-scope-check.md` through `specs/gates/verify-5-gaps.md`. Layer 0 (scope check) runs against each work item's diff individually; layers 1–5 run against the merged patch (all work item diffs combined). When `/shit:verify` is run interactively against a whole project rather than a single pipeline, treat the working tree as the merged patch and skip layer 0 (there are no per-work-item diffs in this mode).

The first two are always run. The third is driven by the distilled checks the user has put into the gate files (typically generated by `/shit:distill`).

## Discovering the Project's Build System

The plugin has no config file. To learn this project's build, test, and coverage commands, **read the project's `CLAUDE.md` file at the project root**. It should describe:

- How to build the project
- How to run the unit/integration test suite
- How to produce a coverage report
- Any per-language linters or static analysis the project uses

If `CLAUDE.md` does not exist or does not describe these things, stop and tell the user. Do not guess at build commands.

## Execution Plan

Maximize parallelism. Launch all independent work streams simultaneously, then collect results.

**IMPORTANT:** Use `run_in_background` for long-running bash commands. Execute fast checks inline while background tasks run. Read pre-generated reports from disk when available.

### Phase 1: Launch Background Streams + Inline Checks

**In a single message, launch these in parallel:**

1. **Stream A — Build** (background bash):
   Run the project's build command (from CLAUDE.md). Capture stderr+stdout. Save output for Section 1 (warning analysis).

2. **Stream B — Tests + Coverage** (background bash):
   Run the project's test command with coverage enabled (from CLAUDE.md). Capture output. Save for Sections 2 (test results) and 3 (coverage).

3. **Stream C — Attestation** (read from disk, fallback to Agent):
   Read today's attestation report from `specs/4-docs/attestation-report_{YYYY-MM-DD}.md`.
   If the file exists, extract the executive summary table and top gaps for Section 4.
   If the file does not exist, launch a general-purpose Agent subagent with prompt: `Run the /shit:attest-report slash command. When complete, read the generated attestation-report file and return the executive summary table and top 10 gaps.` This is the heaviest operation — runs in background.

**While background streams run, execute these inline (they're fast — seconds each):**

4. **Stream D — Provenance Coverage** (Section 6, read from disk or inline):
   If `specs/4-docs/spec-status-report_{YYYY-MM-DD}.md` exists (today's date), read it and extract per-module coverage data for Section 6.
   If the file does not exist, fall back to inline provenance scanning:
   - Read spec module index `specs/2-spec/000-index.md` to get requirement IDs per module
   - Scan project source for any provenance marker pattern (`@provenance:`, `MARK: - Provenance:`, etc.)
   - Scan project tests for the same patterns AND test framework tags
   - Compute per-module implementation% and test%

5. **Stream E — Pipeline Traceability** (Section 5, read from disk or inline):
   If `specs/4-docs/audit-spec-report_{YYYY-MM-DD}.md` exists (today's date), read it and extract PRD coverage %, module health, and orphaned provenance data for Section 5.
   If the file does not exist, fall back to inline computation depending on Stream D.

6. **Stream F — Informational Reports** (Section 9):

   **9a. TODO/FIXME/HACK:**
   - Scan project source and test files for `TODO|FIXME|HACK` (case-insensitive)
   - List each with file:line and comment text

   **9b. Unwired code analysis (exhaustive — do NOT skip or sample):**

   **Why this step matters — read this before executing:** This is one of the most important checks in the entire suite. In agent-driven development, a common failure mode is that an agent implements a feature — writes the type, the methods, the tests — but doesn't complete the wiring. The session ends, the code passes review because it compiles and tests pass, but the feature silently sits disconnected. By running this analysis exhaustively, you are catching what other agents missed. Do this thoroughly — every declaration checked, every unwired finding diagnosed.

   - Find ALL type declarations (classes, structs, interfaces, traits, enums, modules) and function declarations in the project's source files
   - For EACH declaration, search the entire source + test tree for references outside the declaring file
   - "Unwired" means the declaration exists but has no callers/references outside its own file
   - For each unwired declaration, diagnose **why** it's unwired and classify as:
     - **Development gap** — code written for a purpose that hasn't been connected yet (wiring incomplete, not the code)
     - **Obsolete** — code superseded by a different approach, can be removed
     - **Premature API** — public surface declared speculatively with no consumer yet
   - Label all results as heuristic — false positives expected (entry points, protocol witnesses, dynamic dispatch, generics)
   - **This analysis must be exhaustive.** Every declaration is checked. No sampling, no shortcuts.

   **9c. Dependency freshness:**
   Check the project's package manifest and run the appropriate "list outdated" command for the project's package manager (read CLAUDE.md to know which one). Then check latest releases.

   **9d. File complexity:**
   List the largest source files by line count and the highest function-count files.

7. **Stream G — Gate-Defined Checks** (Sections 7 and 8):

   For each gate file in `specs/gates/verify-*.md`, read the file and extract the distilled checks from the `<!-- BEGIN DISTILLED -->` / `<!-- END DISTILLED -->` block. Each distilled check is an instruction; execute it as written.

   Distilled checks are language-agnostic descriptions of what to verify. Pick the appropriate mechanism for each — file scanning, code reading, running a tool, checking a report. Report results per check.

   Group results by gate file:

   **Section 7 — Verification Layers (gate-defined):**
   - 7a. Layer 0 — Scope Check (`gates/verify-0-scope-check.md`) — runs per work item, skipped in whole-project mode
   - 7b. Layer 1 — Traceability (`gates/verify-1-traceability.md`) — runs against merged patch
   - 7c. Layer 2 — Invariants (`gates/verify-2-invariants.md`)
   - 7d. Layer 3 — Architecture (`gates/verify-3-architecture.md`)
   - 7e. Layer 4 — Blast Radius (`gates/verify-4-blast-radius.md`)
   - 7f. Layer 5 — Gaps (`gates/verify-5-gaps.md`)

   **Section 8 — Phase Gates (gate-defined):**
   - 8a. Design (`gates/design.md`)
   - 8b. Breakdown (`gates/breakdown.md`)

   If a gate file has no distilled checks (the BEGIN DISTILLED block is empty), report it as "no project-specific checks defined — run `/shit:distill` to populate."

### Phase 2: Collect Background Results

After Streams A, B, C complete:

1. **Parse build output** (Section 1):
   - Look for warning lines in captured output
   - Count and categorize

2. **Parse test output** (Section 2):
   - Extract total/passed/failed/skipped from test summary
   - Extract any failure details

3. **Parse coverage** (Section 3):
   The coverage report format depends on the project's tooling. The CLAUDE.md should describe how to interpret it. Typically: total coverage %, per-file coverage, per-directory rollups. Build a hierarchical filesystem tree if the data is granular enough.

4. **Read attestation results** (Section 4):
   Extract executive summary and top gaps from the attestation report.

Compile the full report at `specs/4-docs/verification-report_{YYYY-MM-DD}.md` using this template:

**Display the Executive Summary table** in the conversation after writing the file.

## Notes

- The plugin's `/verify` is intentionally less prescriptive than typical CI. The structural rules that matter to your project live in `specs/gates/verify-*.md`, not hardcoded into this command. If you want a particular check enforced, distill it into the appropriate gate file.
- Universal checks (build, tests, coverage, TODO scan, unwired analysis) always run. They are not gated on `specs/gates/` content.
- This command is read-only with respect to the project. It writes only to `specs/4-docs/`.
