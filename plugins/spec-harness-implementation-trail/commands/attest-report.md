# Full Attestation Report — Parallel Sweep

Run `/shit:attest` across all spec modules in parallel, then compile results into a single attestation report.

## Process

### 1. Discover Active Modules

Read `specs/2-spec/000-index.md` and extract all modules with status `complete` and active requirement count > 0. Skip stubs. Build a list of module numbers and names.

### 2. Spin Up Parallel Workers

For each active module, launch a parallel Agent subagent. Each worker's assignment is to run the `/shit:attest` slash command on its assigned module number.

**Worker prompt template:**

```
Run the /shit:attest slash command with argument: {NNN}

This will analyze spec module {NNN}-{name} and write a report to specs/4-docs/attestations/attest-{NNN}-{name}.md.

When done, report back the module verdict line (N conformant, N partial, N non-conformant, N not assessed).
```

**Worker properties:**

- Workers are read-only analysts — they read spec, code, and test files but only write to `specs/4-docs/attestations/`. No merge coordination needed.
- All workers can run in parallel since they write to separate output files and don't modify source code.
- Prefer a fast, analytical model for workers — attestation is reading and reasoning, not creative work.

### 3. Monitor and Collect

As workers complete, collect their module verdict summaries. Track progress: N/total modules complete.

### 4. Compile Combined Report

Once all workers have finished, read all individual attestation files from `specs/4-docs/attestations/attest-*.md` and compile them into a single combined report at:

```
specs/4-docs/attestation-report_{YYYY-MM-DD}.md
```

The combined report has this structure:

```markdown
# Attestation Report — {date}

**Scope:** Full specification ({N} modules, {N} active requirements)
**Generated by:** Parallel attestation ({N} workers)

## Executive Summary

| Verdict | Count | % |
|---------|-------|---|
| CONFORMANT | N | X% |
| PARTIAL | N | X% |
| NON-CONFORMANT | N | X% |
| NOT ASSESSED | N | X% |

## Module Verdicts

| Module | Active Reqs | Conformant | Partial | Non-Conformant | Not Assessed |
|--------|-------------|------------|---------|----------------|--------------|
| 002-invariants | 9 | ... | ... | ... | ... |
...

## Top Gaps (Highest Impact)

Aggregate all gaps from individual attestations. Sort by priority (must-have first), then by module. List the top 20 most impactful gaps — those where must-have requirements have unsatisfied properties or unverified assertions.

## Per-Module Detail

(Include each individual module attestation below, in module number order)

---

### Module 002 — Invariants

(paste content from attest-002-invariants.md)

---

### Module 003 — System Architecture

(paste content from attest-003-system-architecture.md)

...
```

### 5. .gitignore note

The individual per-module files in `specs/4-docs/attestations/` are build artifacts. The user should add that directory to their `.gitignore` if they don't want them tracked. Only the combined dated report is meant to be committed (and even that is optional — it can be regenerated).

## Output

Write the combined report to `specs/4-docs/attestation-report_{YYYY-MM-DD}.md` and display the executive summary in the conversation.
